
基于 React + Redux 设计并实现智能客服前端模块，构建统一的会话与消息状态管理体系；支持访客端 UI 
配置化定制（主题、尺寸、功能开关），并维护图片 / 文件 / 音视频等多媒体消息能力，保障复杂交互场景下的稳定性与性能。


Q: 你简历中提到：
「使用 React + Redux 编写智能客服功能模块，支持访客端 UI 定制（主题、尺寸、音视频功能）及多媒体消息后台系统维护」
能先整体介绍一下这个智能客服模块的架构和你负责的部分吗？
A: 这个智能客服主要面向 C 端访客，提供文本、图片、音视频等多种消息能力，并支持企业自定义客服窗口样式。
前端整体基于 React + Redux，按功能拆成 会话区、消息输入区、客服状态区、配置中心 等模块。
我主要负责 访客端 UI 定制体系设计、消息状态管理，以及多媒体消息模块的维护和优化，
包括主题切换、窗口尺寸适配、音视频能力的接入等。

Q: 为什么这个项目选择 Redux？你们的 state 是如何设计的？
A: 客服系统中有大量跨组件共享状态，比如：
当前会话信息（sessionId、访客信息）
消息列表（文本、图片、音视频、系统消息）
客服在线状态 / 转接状态
UI 配置（主题、尺寸、功能开关）
这些状态变化频繁且存在异步更新，使用 Redux 能统一管理数据流，避免 props 层层传递。
state 设计上我会按领域拆分，比如：
chatSession
messageStore
uiConfig
mediaState

Q: 你提到支持访客端 UI 定制，具体是怎么实现的？是写死配置还是动态配置？
A: UI 定制是通过一套配置驱动方案实现的。
后台会下发一份 UI 配置 JSON，包括：
主题色（主色、按钮色）
窗口尺寸（宽高、是否浮窗）
功能开关（是否支持音视频、是否显示表情）
前端在初始化时将配置存入 Redux 的 uiConfig，
比如主题色直接映射到 CSS 变量，尺寸通过 layout 组件计算。

Q: 音视频功能是怎么做的？用的是什么方案？
A: 音视频主要基于 WebRTC，前端负责：
媒体设备权限申请
本地流采集与播放
信令状态的 UI 管理
实际信令是由后端和第三方服务完成，前端主要通过 Redux 维护音视频的状态机，比如：
calling
connected
muted
ended

Q: 多媒体消息在前端是怎么处理的？有没有遇到性能或稳定性问题？
A: 多媒体消息主要包括图片、文件和音视频消息。
前端会对消息类型做统一抽象，每条消息都有：
id
type
status（sending / success / fail）
后台给企业定制消息，包括UI侧触发多媒体消息的命令，服务端下发多媒体消息，前端渲染

Q： 如果不用 Redux，你觉得还能怎么做？
A： 如果项目规模较小，可以用 React Context + useReducer。
但在客服系统这种高频状态变更、模块多、可扩展性要求高的场景下，Redux 在可维护性和可预测性上更有优势。
